# CoreGuard

**CoreGuard** — 一款安卓应用，用于保护企业网络免受用户设备上不受信任应用的威胁。使用标准 Android 工作配置文件进行隔离，配合基于触发器的自动化和网络 Kill Switch。功能丰富的 Shelter 替代方案。

---

## 核心理念

主要问题：员工在工作设备上安装来自未经验证来源的应用。这些应用可能扫描企业网络、收集连接数据并将信息传输给第三方。

**解决方案：** 将不受信任的应用放入隔离的 Android 工作配置文件中。一台设备上的两个独立空间：

```
┌─────────────────────────┐   ┌─────────────────────────┐
│   主配置文件             │   │   工作配置文件            │
│   企业资源               │   │   不受信任的              │
│   受保护的连接           │   │   应用程序                │
│   工作工具               │   │   ← 与网络隔离           │
└─────────────────────────┘   └─────────────────────────┘
         ↑ 不同的 uid、不同的数据、不同的网络栈
```

不同配置文件中的应用存在于不同的 Linux uid 命名空间中。它们无法看到彼此的数据和网络状态。工作配置文件中的应用**无法访问**主配置文件的企业资源和连接。

CoreGuard 实现自动化管理：当触发器激活时，不受信任的应用被冻结并完全断网。

---

## 保护机制：工作配置文件中的应用隔离

### 基本规则：不受信任的应用 — 仅在工作配置文件中

为使保护生效，遵循一条规则：**通过工作配置文件的 Play Store 安装不受信任的应用**。这在内核级别创建隔离。

两个配置文件**同时并行运行**。工作配置文件应用无法看到主配置文件的连接和数据。

### 应用在工作配置文件内看到什么

工作配置文件是一个独立的 Linux uid 容器。`ConnectivityManager` 按 userId 过滤网络 — WP 流量与主配置文件完全隔离。

关键保护 — **进程挂起**：当触发器激活时，所有不受信任的应用被设置为 `setPackagesSuspended` — 死进程无法检查或传输任何内容。

| 应用在 WP 中检查的内容 | 结果 | 备注 |
|---|:---:|---|
| `ConnectivityManager` — 网络类型 | ✅ 隔离 | 只能看到自己的网络 |
| `NetworkCapabilities` — 传输方式 | ✅ 隔离 | 无主配置文件连接数据 |
| DNS 服务器 | ✅ 隔离 | 自己的 DNS，非企业 DNS |
| 网络接口 | ✅ 冻结时 | 触发器激活时应用**被挂起** |
| IP 地址 | ✅ 隔离 | 直接 IP，无法访问企业隧道 |
| `System.getProperty("http.proxyHost")` | ✅ | `null` |

### 触发器 — 自动控制

CoreGuard 实现 **Zero Trust** 方法：当设备可以访问企业网络时，不受信任的应用不得运行。组合触发系统对网络状态变化做出反应：

- VPN、Wi-Fi、移动网络、运营商 — 4 种触发器类型
- 企业 Wi-Fi 连接 — 自动 Kill Switch（最大保护）
- 任何触发器 → 立即冻结不受信任的应用

## 为什么不只用 Shelter？

Shelter 可以创建工作配置文件并克隆应用。但冻结应用只能**手动**进行。没有自动化，没有 Kill Switch。

CoreGuard 做同样的事，**再加上**：

| 功能 | Shelter | CoreGuard |
|---|:---:|:---:|
| 工作配置文件（应用隔离） | ✅ | ✅ |
| 手动冻结/解冻 | ✅ | ✅ |
| **按触发器自动冻结** | ❌ | ✅ |
| **网络 Kill Switch（完全断网）** | ❌ | ✅ |
| **企业 Wi-Fi → 自动 Kill Switch** | ❌ | ✅ |
| **触发器：虚拟/企业网络** | ❌ | ✅ |
| **触发器：Wi-Fi SSID** | ❌ | ✅ |
| **触发器：移动网络** | ❌ | ✅ |
| **触发器：移动运营商** | ❌ | ✅ |
| 配置文件间剪贴板共享 | ❌ | ✅ |
| 配置文件间联系人共享 | ❌ | ✅ |
| **应用权限控制（位置、相机、麦克风…）** | ❌ | ✅ |
| **按应用权限设置** | ❌ | ✅ |
| **阻止从工作配置文件卸载应用** | ❌ | ✅ |

## 工作原理

### 工作配置文件 — 两个独立空间

工作配置文件通过标准 Android 系统对话框创建。无需 ADB、恢复出厂设置或 root。创建后，设备上出现两个完全隔离的容器：

- **主配置文件** — 企业资源、受保护连接、工作工具。
- **工作配置文件** — 不受信任应用的独立 uid 空间。

数据、帐户、缓存、SharedPreferences — 一切在内核级别分离。

### 触发器 — 自动切换

用户配置触发条件。当任何触发器激活时：

- **0 毫秒** — Kill Switch：工作配置文件上的本地 TUN 黑洞丢弃所有流量 → 不受信任应用的任何字节都不会离开
- **100 毫秒** — `setPackagesSuspended()`：所有 WP 应用被冻结（灰色图标，无 CPU / RAM / 网络 / 推送通知）

当所有触发器都不活跃时 — 一切自动恢复。

### Kill Switch — 本地防火墙

Kill Switch 是一个本地 TUN 接口，路由为 `0.0.0.0/0`，读取所有数据包并丢弃。没有任何字节离开设备。在工作配置文件内运行，不影响主配置文件。

Kill Switch 是**可选的** — 在设置中单独启用。默认关闭。

## 触发器

| 触发器 | 描述 | 默认 |
|---|---|:---:|
| 🔒 **虚拟网络** | 设备上有活跃的虚拟/企业网络 | ✅ 开启 |
| 📶 **Wi-Fi SSID** | 连接到用户列表中的网络 | ❌ 关闭 |
| 🏢 **企业 Wi-Fi → Kill Switch** | Wi-Fi 匹配时自动启用 Kill Switch | ❌ 关闭 |
| 📱 **移动网络** | 通过移动数据的任何连接 | ✅ 开启 |
| 📡 **移动运营商** | 连接到特定运营商 | ❌ 关闭 |

逻辑始终为 **OR** — 一个触发条件就足够。

### 网络策略

不同连接类型可以激活不同保护级别：

- **企业 Wi-Fi**（列表中的 SSID + Kill Switch 标志）→ **最大保护**：Kill Switch 阻止所有网络，所有标记的应用被冻结
- **VPN / 移动网络** → **标准保护**：冻结标记的应用，其他应用正常工作
- **移动运营商** → **可配置保护**：响应特定运营商（如企业 SIM 卡）

## 系统要求

- Android 8.0+（API 26）
- 支持工作配置文件的设备（99% 的设备）
- Wi-Fi SSID 触发器需要：位置权限（Android 要求）

**开发需要：** JDK 17、Android SDK（API 26+）、ADB。

---

## 构建

```powershell
# 从项目根目录
.\gradlew.bat assembleDebug
```

APK 位置：`app/build/outputs/apk/debug/app-debug.apk`

**安装到设备（工作配置文件）：**
```powershell
adb install --user all -r app\build\outputs\apk\debug\app-debug.apk
```
> `--user all` 标志是必须的 — 否则只更新主配置文件（user 0）。
